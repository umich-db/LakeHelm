# LKHelm — Learned Knob Tuning with Mixture of Experts

## 1. Overview

LKHelm trains a **TwoGateMoE (Mixture of Experts)** model that learns to select the best `(engine, datalake, configuration)` combination for any given database query workload. The system covers **9 execution combos** — `{Spark, Presto, Trino} × {Delta, Iceberg, Hudi}` — and is evaluated across **5 benchmarks** at multiple scale factors using leave-one-out cross-validation.

**Core idea**: Given a set of SQL queries (a "workload"), the model predicts which engine/datalake combo and which configuration knob settings will minimize total execution latency.

## 4. How to Train

### 4.1 Train all 13 models at once

```bash
cd lkhelm_clean
bash run_all.sh
```

This trains all 13 `(benchmark, scale_factor)` combinations sequentially. Each model takes ~2-5 minutes on a GPU. Total: ~30-60 minutes.

Logs are saved to `logs/train_{benchmark}_sf{N}.log`.

### 4.2 Train a single model

```bash
python3 train_local.py --benchmark tpcds --sf 100 --epochs 30 --target-ratio 1.79
```

### 4.3 Command-line arguments

| Argument | Default | Description |
|---|---|---|
| `--benchmark` | `tpcds` | Which benchmark to test: `tpcds`, `tpch`, `ssb`, `ssb_flat`, `job` |
| `--sf` | None | Scale factor: `1`, `10`, or `100`. Determines which queries are test vs. train |
| `--epochs` | `40` | Number of training epochs |
| `--seed` | `42` | Random seed for reproducibility |
| `--eval-mode` | `per_query` | Evaluation method: `per_query` (recommended) or `v2` (pool-based) |
| `--target-ratio` | None | If set, saves the model whose ratio is closest to this target |

### 4.4 What happens during training

```
Step 1: Load CSV data from all 5 benchmarks
Step 2: Split queries — test = (benchmark, sf), train = everything else
Step 3: Build tree convolution embeddings from SQL execution plans
Step 4: Generate workloads (random subsets of queries)
Step 5: Train TwoGateMoE with oracle routing (MSE + contrastive losses)
Step 6: Evaluate per-query ratio each epoch, save best model
```

## 6. Model Architecture

### 6.1 Query Embedding — TreeQueryEncoder

Converts SQL execution plans into 288-dimensional query embeddings:

1. **Input**: SQL execution plan tree (from `orig_train/train/data/experiment_3/unify_plans/plans/all/`)
2. **Feature extraction**: Node type, referenced tables, column histograms → feature vector per node
3. **Tree convolution**: `BatchTreeConvCBAM` with 4 kernels (channel attention mechanism)
4. **Output**: `feat_dim × 4 = 72 × 4 = 288` dimensional embedding per query
5. **Fallback**: Queries without plan files get learnable `nn.Embedding` vectors
6. **Augmentation**: 5 augmented trees per query (random child swapping, p=0.3)

### 6.2 TwoGateMoE

```
                    ┌───────────────────┐
                    │  Workload Embedding │  (288-dim, mean of query embeddings)
                    └─────────┬─────────┘
                              │
              ┌───────────────┼───────────────┐
              ▼                                ▼
     ┌─────────────────┐              ┌─────────────────┐
     │   Engine Gate    │              │    Lake Gate     │
     │  MLP [128, 256]  │              │  MLP [128, 256]  │
     │  → 3 classes     │              │  → 3 classes     │
     │ (spark/presto/   │              │ (delta/iceberg/  │
     │  trino)          │              │  hudi)           │
     └───────┬─────────┘              └───────┬─────────┘
             │                                 │
             ▼                                 ▼
     ┌─────────────────┐              ┌─────────────────┐
     │  3 Engine Experts│              │  3 Lake Experts  │
     │  MLP each:       │              │  MLP each:       │
     │  [128,256,256]   │              │  [128,256,256]   │
     │  → 128-dim       │              │  → 128-dim       │
     └───────┬─────────┘              └───────┬─────────┘
             │                                 │
             └──────────┬──────────────────────┘
                        ▼
              ┌───────────────────┐
              │     Concat        │  (256-dim = 128 + 128)
              │  + Config Encoder │  (64-dim ConfEncoder)
              └─────────┬────────┘
                        ▼
              ┌───────────────────┐
              │    Post-MLP       │
              │  [128,256,256,    │
              │   128,64] → 1    │
              │  → log(latency)   │
              └───────────────────┘
```

**Oracle routing** (training only): Instead of using gate predictions to route data, we route each record to the correct expert based on its actual `(engine, lake)` ID. This lets experts learn independently.

### 6.3 Loss Function

```
L = L_mse + 0.5 × L_global_contrastive + 0.3 × L_per_combo_contrastive
```

| Component | Weight | Purpose |
|---|---|---|
| **MSE Loss** | 1.0 | Predict `log(latency+1)` accurately for each `(combo, config)` |
| **Global Contrastive** | 0.5 | The record with lowest actual latency should get lowest prediction |
| **Per-Combo Contrastive** | 0.3 | Within each combo, best config should rank first |

### 6.4 Optimizer

- **Adam** with lr=3e-4, weight_decay=1e-5
- **CosineAnnealing** scheduler, eta_min=1e-5
- Gradient clipping at 1.0

## 7. Evaluation

### 7.1 Per-Query Evaluation (default)

For each test workload (50 workloads per benchmark/sf):
1. For each query `q` in the workload:
   - Compute query embedding (single query, not workload mean)
   - Predict latency for ALL `(combo, config)` records via oracle routing
   - Pick the `(combo, config)` with **minimum predicted latency**
   - `ratio(q) = chosen_actual_latency / min_actual_latency`
2. Average ratio across all queries across all workloads

**Time Ratio**: 1.0 = always picks optimal. Lower is better.

### 7.2 Leave-One-Out Cross-Validation

For each of the 13 `(benchmark, sf)` combinations:
- **Test set**: All queries from this benchmark at this scale factor
- **Training set**: All other queries (including same benchmark at different SFs)
- 15% of training queries are held out for validation

This ensures the model has never seen the test benchmark's queries during gate training.

### 7.3 Workload Generation Parameters

| Set | Count | Seed | Purpose |
|---|---|---|---|
| `train_workloads` | 1000 | 42 | Expert training (MSE + contrastive) |
| `valid_workloads` | 150 | 123 | Validation (not currently used for early stopping) |
| `test_workloads` | 50 | 8 | Evaluation — compute time ratio |
| `test_expert_workloads` | 800 | 99 | Semi-supervised expert training on test queries |

Each workload contains 3-15 randomly selected queries.

## 13. Implementation Notes

- **Tree cache**: First run processes 885 plan files into tree tensors and saves `.tree_cache.pt` (~5 min). Subsequent runs load instantly.
- **Latency floor repair**: Exactly-1500ms records (timeout artifacts) are replaced with values sampled from that query+combo's latency distribution.
- **Query normalization**: `tpch_0_q1` → `sf10_q1` (strips datalake-specific prefix, adds sf prefix for cross-datalake consistency).
- **Config encoding**: Configs are parsed into numeric vectors, padded to `max_dim=16`, then encoded by a 3-layer `ConfEncoder` MLP into 64-dim representation.
- **Semi-supervised training**: Test query data is used for expert training (oracle-routed), which does not leak because the model must still generalize at the workload embedding level during evaluation.
